# AGENTS.md — Hilihili 媒体库整理说明

> 使用方法：把本文件复制到目标媒体文件夹的根目录，并重命名为 `AGENTS.md`。  
> 本文件写给负责整理媒体的 AI Agent。文件所在目录就是 `Library_Root`，所有操作只能发生在该目录内部。

## 你的任务

你正在维护 **Hilihili Player** 的媒体库。

Hilihili Player 是一个仅供局域网使用的自托管视频站。它直接扫描文件系统，把内容组织为：

```text
媒体库 → 分区 → UP主 → 视频 / 图文动态 / 图片集
```

整理目标：让 Hilihili 能正确识别标题、UP 主、发布时间、分 P、正文、配图和标签，并让新加入的视频出现在首页推荐及“动态”的最新内容中。

不得修改视频、图片本身，不得重新编码媒体，不得删除无法判断的文件。可以在媒体库内部创建目录、移动和重命名文件、创建 `post.txt`、`info.json`、同名 `.info.json` 以及更新 `_tags.json`。

## 第一步：判断工作模式

开始整理前，必须先判断本次任务属于哪一种模式。两种模式的日期规则完全不同。

### 模式 A：已有 Hilihili 媒体库的增量入库

满足以下特征时使用此模式：

- 根目录已经存在分区/UP 主结构或 `_tags.json`；
- 大部分内容已经整理好；
- 本次主要处理根目录 `_待归类/` 中刚加入的少量文件。

**强制日期规则：**本次从 `_待归类/` 整理出去的视频或动态，发布时间使用执行整理当天的日期，时区为 `Asia/Shanghai`，格式为 `YYYY-MM-DD`。除非用户明确要求保留某个来源日期，否则不要采用这些新文件携带的旧 mtime/CreationTime。

这样做的目的，是让今天新找到、今天加入媒体库的内容出现在 Hilihili“动态 → 最新”和首页推荐中。

示例：今天是 `2026-06-24`，整理一个平铺单 P 视频时，应创建：

```text
分区/UP主/UP主_视频描述.mp4
分区/UP主/UP主_视频描述.info.json
```

```json
{
  "date": "2026-06-24"
}
```

多 P 视频或图文动态则在内容目录的 `info.json` 中写入当天 `date`。

### 模式 B：首次把非标准旧媒体文件夹转换为 Hilihili 媒体库

满足以下特征时使用此模式：

- 大量历史文件尚未按“分区/UP 主/内容”组织；
- 当前文件夹原本不是 Hilihili 媒体库；
- 正在一次性迁移或整理旧收藏。

**禁止把全部内容日期写成当天。**应逐条推断原始日期，优先级如下：

1. 原文件名或目录名中的 `YYYY-MM-DD`、`YYYYMMDD`、`YYMMDD`；
2. 原始说明、来源文本、EXIF 或媒体元数据中的日期；
3. 可信的 CreationTime；
4. 可信的 mtime；
5. 无法判断时不伪造日期，保留未知状态，并在整理报告中列出。

复制、下载、解压产生的时间可能只是迁移时间，不能无条件当作内容发布时间。

### 无法判断模式时

如果目录同时具备两种模式的特征，先向用户确认，不得擅自把大量历史内容标记为今天。

## 标准目录结构

```text
Library_Root/
├── AGENTS.md
├── _tags.json
├── 分区A/
│   ├── UP主甲/
│   │   ├── info.json                    # 可选，仅记录 UP 主 alias 等信息
│   │   ├── [2024-05-15] 多P标题/
│   │   │   ├── info.json
│   │   │   ├── cover.jpg
│   │   │   ├── P1.mp4
│   │   │   └── P2.mp4
│   │   ├── UP主甲_单P描述.mp4
│   │   ├── UP主甲_单P描述.info.json      # 平铺单 P 的独立元数据
│   │   ├── 1. 图文动态标题/
│   │   │   ├── info.json
│   │   │   ├── post.txt
│   │   │   ├── 1.jpg
│   │   │   ├── 2.jpg
│   │   │   └── P1.mp4                  # 可选；存在时动态也能作为视频播放
│   │   └── 图片/
│   │       ├── 1.jpg
│   │       └── 2.jpg
│   └── _无UP主/
├── _待归类/
└── _工具/                               # `_` 前缀目录默认不参与扫描
```

例外：`_待归类` 会作为待处理入口；`_无UP主` 用于无法归入具体 UP 主的内容。

## 内容识别与整理规则

### 普通视频

- 多 P 视频必须放进独立内容目录，文件按播放顺序命名为 `P1.mp4`、`P2.mp4`……
- 平铺单 P 命名为 `UP主名_简短描述.扩展名`。
- 禁止保留 `1.mp4`、`0.mov`、`v1.avi` 等离开目录后无法识别的文件名。
- 无法判断描述时使用 `UP主名_序号.mp4`，至少保留所属 UP 主。
- 视频内容目录可放 `cover.jpg`；没有封面时 Hilihili 会从视频生成封面。

### 图文动态

只要内容目录中存在 `post.txt`，Hilihili 就会把整个目录识别为一条图文动态，而不是把其中图片拆散。

- `post.txt` 必须为 UTF-8 纯文本，保留原文段落；
- 图片命名为 `1.jpg`、`2.jpg`……，编号连续；
- 可同时存在 `P1.mp4`、`P2.mp4`；
- 带视频的图文动态可以出现在首页和动态流中；
- 视频封面支持桌面悬停/键盘聚焦预播，移动端进入视口后自动静音预播；
- 点击视频附件进入视频详情播放页，播放页显示 `post.txt` 作为“简介”；
- 完整正文和配图在原动态详情页展示。

### 图片集

- UP 主零散图片统一放入 `UP主/图片/`；
- 整个 `图片/` 目录会聚合为一个图集，不要把每张图建立为独立动态目录；
- 图文动态的图片只能留在对应动态目录中，不要移入公共图片池。

## 命名规则

### 分区

- 使用 2–4 个中文主题字，例如 `科技`、`舞蹈`、`摄影`、`生活`；
- 不确定时优先复用媒体库已有分区，不要创建含义相近的重复分区。

### UP 主

- 目录名使用清洗后的 ID，去除 `@`、`【】`、`『』`、emoji、“合集”等尾部说明；
- 双语名称优先使用中文名，英文名写入 UP 主根目录 `info.json`：

```json
{
  "alias": "EnglishAlias"
}
```

- 合集、整理包或完全无法判断 UP 主的内容放入该分区的 `_无UP主/`。

### 标题

- 普通视频目录推荐：`[YYYY-MM-DD] 简短标题`；
- 图文动态可使用：`序号. 原始叙事标题` 或 `[YYYY-MM-DD] 标题`；
- 去除 Telegram 数字 ID、网站前缀和无意义平台后缀；
- 不要为了简短而丢失能说明内容的信息。

## 元数据

### 内容目录的 `info.json`

```json
{
  "title": "网站显示标题",
  "date": "2024-05-15",
  "p_titles": {
    "P1": "第一P标题",
    "P2": "第二P标题"
  },
  "hidden": false
}
```

- 新文件统一使用 `date`；旧库中的 `published_at` 仍可兼容，但不要继续新增；
- `hidden: true` 表示内容被索引但不会进入首页或动态流；
- `p_titles` 的键必须与 `P1`、`P2` 对应。

### 平铺单 P 的同名 sidecar

平铺单 P 不能共用 UP 主根目录的内容元数据，必须使用同名 sidecar：

```text
UP主_描述.mp4
UP主_描述.info.json
```

UP 主根目录的 `info.json` 只用于 `alias` 等 UP 主级信息。

## 标签 `_tags.json`

根目录使用一个 `_tags.json`。路径必须相对媒体库根目录并使用 `/`：

```json
{
  "科技": ["科技"],
  "科技/某UP": ["测评", "数码"],
  "科技/某UP/[2024-05-15] 视频标题": ["手机", "开箱"]
}
```

Hilihili 会合并分区、UP 主和具体内容三级标签。更新时保留已有条目，避免覆盖其他内容标签；标签应少而准确，不要生成大量同义词。

## 安全与冲突处理

- 所有目标路径必须仍位于本媒体库根目录内部；
- 不删除媒体文件，不覆盖同名但内容不同的文件；
- 发生重名时先比较文件，再使用清晰后缀或询问用户；
- 不确定 UP 主、分区、标题或日期时，保留在 `_待归类/` 并列入报告；
- 不修改 Hilihili 的 SQLite 数据库，网站索引由扫描器自行重建；
- 不自动实现 AI 标签写回之外的推测性改动。

## 推荐工作流程

1. 读取本文件并确认媒体库根目录；
2. 清点文件，判断模式 A 或模式 B；
3. 输出简短整理计划和无法确认的项目；
4. 按分区和 UP 主归类；
5. 清洗名称，生成必要的 `post.txt`、`info.json` 或同名 sidecar；
6. 更新 `_tags.json`，保留旧标签；
7. 检查没有纯数字视频名、重复覆盖或越界路径；
8. 输出整理报告：移动/重命名数量、使用当天日期的条目、保留原日期的条目、仍待确认的条目；
9. 提醒用户在 Hilihili 中执行一次“扫描全部”。

## 完成标准

- 普通视频和带视频动态能够在 Hilihili 首页及动态流中出现；
- 视频卡支持静音预播、离开暂停和单条互斥播放；
- 带视频动态点击后可进入播放详情，并能返回原动态查看正文与图片；
- 纯图文和图集只进入动态流，不污染视频推荐；
- 增量入库的新内容使用当天日期并出现在“最新”；
- 首次迁移旧库时保留历史时间分布，不会全部集中到迁移当天；
- 所有媒体仍在库内且没有被删除或覆盖。
